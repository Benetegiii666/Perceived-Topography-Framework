# RP_001: Optimize for Framework Failure, Not Framework Confirmation

**Status:** Research Protocol Candidate
**Date:** 2026-06-10

## Observation

As the framework begins encountering real-world evidence, confirmations become less valuable than disconfirmations.

Repeated examples that fit existing categories increase confidence but do not significantly improve architecture. Failures that do not fit existing categories create opportunities for discovery and refinement.

## Principle

When evaluating a real-world event:

Do **not** begin by asking: *"Which category does this fit?"*

Begin by asking: **"Is there a plausible explanation that does not fit the current categories?"**

## Purpose

- Prevent category lock-in
- Protect against unfalsifiability (CA_005)
- Preserve the framework's ability to evolve when reality disagrees

## Operational Question

For any investigated failure:

1. Goal Failure?
2. Context Failure?
3. Attention Failure?
4. Interpretation Failure?
5. **Or: None of the Above?**

Always ask #5 first.

## Expected Outcome

If a failure cannot be reasonably classified, the result is not necessarily a framework failure. It may indicate:

- Missing category
- Missing mechanism
- Incorrect architecture
- Previously unseen phenomenon

Each of these is more valuable to the framework than a clean classification would have been.
