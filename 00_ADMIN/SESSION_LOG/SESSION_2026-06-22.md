## Key Design Decisions Made

1. **Design stance:** Design from reasoning-state preservation outward — not from model choice, vendor choice, chatbot UI, or generic RAG.
2. **Decision State dual-status:** Separate `reasoning_status` (Draft/Provisional/Validated) from `decision_status` (Active/Archived/Superseded) because a decision can be operationally active while its reasoning is only provisional.
3. **Requester confirmation scope:** Requesters can confirm intent and local workflow context but cannot validate domain claims, compliance, evidence, policy, MUOs, or high-consequence premises without holding the relevant authority role.
4. **Information Surface authority taxonomy:** Aligned with Build Discovery Guide trust model (binding/advisory/background/anecdotal).
5. **Confirmation fatigue mitigation:** Validated reasoning earns reduced burden (lightweight applicability acknowledgment), not silent application. Preserves Principle 4.
6. **Martin/Beacon/co-worker:** Treated as compatibility and interface surfaces throughout. Not conceptual authority. MVP recommendation: design for later integration, not integrate now.
