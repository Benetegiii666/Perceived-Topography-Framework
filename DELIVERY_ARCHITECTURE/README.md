# DELIVERY_ARCHITECTURE

## Purpose

Delivery architecture planning for the Perceived Topography Framework implementation. Defines product form factor, application structure, technical delivery options, deployment direction, and visual design backlog.

## What This Folder Is

- Product delivery form decisions (what the thing is, how users interact with it, where it lives)
- Delivery architecture backlog (decisions and design tasks before implementation)
- Visual asset planning (diagrams, site maps, wireframes)

## What This Folder Is Not

- Not the HLD (that lives in `REFERENCE_ARCHITECTURE/`)
- Not the implementation plan (that lives in `IMPLEMENTATION_PLANNING/`)
- Not the field kit (that lives in `FIELD_KITS/`)
- Not implementation code, schemas, or build backlog
- Not final stack selection (delivery assumptions require review)

## How It Relates to Other Artifacts

| Artifact | Relationship |
|---|---|
| HLD v1.0 | Defines conceptual/reference architecture. Delivery architecture translates it into product form. |
| Implementation Slice Plan | Defines first implementation scope. Delivery architecture defines how that scope is delivered. |
| Field Kit | Defines stakeholder discovery process. Delivery architecture may eventually absorb some field kit concepts into product UI. |
| Campaign Brief Workflow Discovery | Defines what users need. Delivery architecture defines how they access it. |
